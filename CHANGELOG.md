## 2026-06-13 · 对峙/选边方向统一 + 立场标签具名（用户实测反馈）

**背景**：上一条给第二幕"选边"做了"靠边站"后，第一幕"对峙"两位主人格仍全宽、方向都偏一边，两幕方向语言不统一；且选边的立场标签写的是抽象"支持 A / 支持 B"，不够直观。

**改了什么**：
1. **方向统一**：对峙发言也按 stance 靠边——A 方靠左、B 方靠右（色条镜像翻到右侧、抬头 `row-reverse`），与选边同一套方向。对峙串行推送、stance 在 `end` 才到，故前端按 `start` 顺序先定方向（首位 A、次位 B）。
2. **主角/配角层级保留**：对峙 `max-width:88%`（更宽、更突出），选边 `80%`（更窄）。
3. **立场具名**：选边标签把"支持 A/B"换成具体人格名（`支持野心家`/`支持守护者`，取 `duelists[0/1]` 名）；对峙发言不再显示立场标签（发言者本身就是那一极）。
4. 去掉 side-b 正文右对齐（对峙是长段落，右对齐别扭），两边正文统一左对齐、只镜像盒子与色条。

**涉及文件**：`static/index.html`（`addSpeech` 分幕打类 + 具名 stance、duel 处理按序定向、`.speech.duel/.debate/.side-*` CSS）。

**验证**：curl 确认 CSS+JS 已 served；视觉需整页强刷走到会议实测。

**状态**：✅ 完成（视觉待整流程实测）。

## 2026-06-13 · 第二幕选边发言"靠边站"（用户实测反馈）

**背景**：第二幕"选边"的 4 个人格发言与第一幕"对峙"两位高分人格用的是**完全相同**的 `.speech` 卡片（`addSpeech` 不分幕），只有 stance 标签不同 → 用户分不清谁是这场会的主轴、谁是来投票的配角。

**改了什么**（方案：把"选边"做成可见的空间位置，呼应阵营天平）：
- 第一幕对峙双方**保持全宽卡片**（左色条、流式蹦字）= 这场会的两极/主角。
- 第二幕选边发言（`addSpeech` 的 `!stream` 分支）按表态加 `debate` + `side-*` 类，整体更窄（80%）、内边距更小：
  - **支持 A** → 靠左（`side-a`）。
  - **支持 B** → 靠右（`side-b`）：色条翻到右边、抬头 `row-reverse`、正文右对齐，与 A 镜像。
  - **第三条路** → 居中（`side-third`）、更窄、**虚线**色条（暗示两边都不站）。

**涉及文件**：`static/index.html`（`addSpeech` 加 `!stream` 分支打类；新增 `.speech.debate/.side-a/.side-b/.side-third` CSS）。

**验证**：curl 确认 CSS+JS 已 served；视觉效果需整页强刷后走到第二幕实测。

**状态**：✅ 完成（视觉待整流程实测）。

## 2026-06-13 · 会前问询加"生成中"占位（用户实测反馈）

**背景**：会前问询屏的两个问题由 `/api/ask` 动态生成（实测 ~4s），生成期间两行只显占位符 `——`、且"让他们开会"按钮可点 → 用户不知道问题还在生成、可能空手就点进会议。

**改了什么**：`goAsk()` 进屏先把两行设为斜体半透明、轻微闪烁的「计算师/本我：正在斟酌要问你什么……」（`.q-pending` 类，`blink` 动画），并 `disabled` 掉"让他们开会"按钮；`/api/ask` 返回后替换成真问题、解禁按钮。用 `try/finally` 兜底，即使生成失败/超时（后端本就有 6s 回退静态文案）也必解除锁定，不卡死。

**涉及文件**：`static/index.html`（`goAsk` 占位+锁定逻辑、新增 `.q-pending` CSS）。

**验证**：curl 确认已 served；`/api/ask` 实测 4.09s，占位态可见约 4 秒。

**状态**：✅ 完成。

## 2026-06-13 · 插话"突兀感"修复（用户实测反馈）

**背景**：用户实测反映地牢插话突兀——立题后直接进 node1，2-3 个没见过的人格带色气泡突然冒出来点评场景，不知道它们是谁、为何说话。根源确认为：① 人格没登场（陌生人）② 场景→声音之间没过渡。

**改了什么**（方案：轻量就地化解，零 LLM、不污染测量）：
1. 立题页加"内阁"引子（lede 下一句"你脑内有一支内阁——六个声音，会一路跟着你…"）。
2. 地牢每节点加"脑内·此刻"眉标；node1 额外一句过渡"你刚踏进这道门，脑海里几个声音同时炸开——"。
3. 人格**首次登场**时气泡带一句人设身份（野心家·脑内的征服者），之后只显名；每人在它首登场的节点亮一次、自然分摊、不重复。身份是"气质"非"答案"，不剧透选项 → 不污染地牢测量。

**涉及文件**：`static/index.html`（PERSONA.role、seenVoices Set、立题 lede、bridge/voices-eyebrow 元素 + CSS、enterNode 逻辑）。

**验证**：浏览器实测——node1 三人格各带身份；node2 过渡句消失、已登场的守护者/本我不重复、仅新出场的共情者显身份 ✓。截图 `docs/screenshots/phase3-intro-crew.png`、`phase3-dungeon-framed.png`。

**状态**：✅ 完成。

## 2026-06-13 · 地牢不再显示测量轴（用户实测反馈）

**背景**：地牢每节点在场景上方显示张力轴标签（"冒进vs谨慎""现实vs理想"），等于把"这道题在测什么"剧透给用户 → 用户会表演（"想显得有理想就选高塔"），污染测量纯度，也破坏冒险沉浸感（像答题分类标签）。

**改了什么**：场景眉标从"测量轴"换成中性"地点名"。`axis` 改为内部元数据、不下发前端。
- `constants.NODES` 每节点加 `place`（废墟深处·将闭的石门 / 废墟旧集市 / 无标记的岔路口），`axis` 保留为内部注释。
- `/api/nodes` 下发 `place`、不再下发 `axis`。
- 前端 `node-axis` 元素改显 `node.place`。

**涉及文件**：`constants.py`、`main.py`（/api/nodes）、`static/index.html`。

**验证**：`/api/nodes` 实测只返回中性地点、无 `axis`、无 "vs" 泄露 ✓。

**状态**：✅ 完成。

## 2026-06-13 · 地牢声音改"匿名"，名字留到会议揭晓（用户实测反馈）

**背景**：用户提出把地牢插话气泡的名字和描述全去掉、只保留颜色。立意：地牢里的声音是**原始、匿名的拉扯**，靠"颜色 + 各自语气"区分；到了**会议**这几个颜色才有名字和脸——形成"先感受、后揭晓"的叙事弧（"原来一路拽我的声音，是我的内心剧组"），也把测量纯度护到极致。

**改了什么**：
- 地牢气泡只剩**彩色正文 + 色边 + 字体**（计算师等宽/造梦师斜体作为第二条非文字线索），去掉名字与描述。
- 随之移除"首次出场身份"（撤回本文件第一条改动的第 3 点）及 `seenVoices` 逻辑。
- **会议保留名字不变**（第一幕"野心家 vs 守护者"、裁决"听野心家的"都需要名字）——名字在会议里才出现 = reveal。
- **首页文案同步**：去掉"内阁/六个声音"（既点数又暗示有名角色），改为"先走一趟试炼——你会听见几个声音替你较劲；走到最后，它们坐下来，把这场没开成的会吵完"——集体、不点名、不点数。
- **每节点集体提示**：过渡句铺到全部三节点（原只 node1），各自贴场景但都带出"几个声音"出现（如 node2"你的手刚碰到那壶水，几个声音就嚷了起来——"）。即"集体提示有声音出现，但不给单个声音贴标签"。

**涉及文件**：`static/index.html`（气泡渲染、移除 seenVoices、BRIDGES 每节点过渡、首页 lede；遗留少量 `.bubble .who` 死 CSS 待扫）。

**待确认**：六色无名是否够区分（缓解：每节点只 2-3 个、颜色差异大、语气迥异）；**会议的"名字揭晓"要让用户能把颜色对上号**——后续验收时重点看这个衔接。

**状态**：✅ 完成（衔接效果待整流程实测）。

## 2026-06-13 · 输入框：单行 → 多行可换行（用户实测反馈）

**背景**：所有输入（议题、自陈两框、裁决补充）是单行 `<input>`，长文本全挤在一行横向滚动；且只有一条下划线、可发现性差。**关键诉求是能换行**。

**改了什么**：四个输入 `<input>` → `<textarea rows=2>`（文本自动折行、可纵向拉伸，JS 仍读 `.value`，无 Enter 提交不受影响）；顺带样式从下划线改成带边框的框（暗 panel 底 + 细边 + 3px 圆角，与场景卡/选项按钮统一），聚焦黄铜描边 + 柔光环。`static/index.html`。

**状态**：✅ 完成（浏览器实测长议题正确折行，截图 `docs/screenshots/intro-input-box.png`）。

## 搁置 · 插话偶尔只出 2/3 条（RPM 限流）

**背景**：用户实测后两个节点常只显 2 个人格的声音。诊断：**全 flash + StepFun RPM=10**——每节点 3 个并行插话，走完三节点累计 9 次调用贴着 10/分钟上限；快速点击/刚跑完会议时，后续节点有 1 条被 429 限流 → `interject_stream` 静默跳过 → 只剩 2 条。干净一跑（有 RPM 余额时）9 条全成。

**根因**：RPM=10 是 **StepFun 账号侧 V0（免费/未充值）档位限制**，非代码可改。充值 ¥100 → V1 → RPM 1000（100×），问题消失。**注意：额度（能花多少）≠ RPM（每分钟几次）；赠送额度通常不计入充值档位，不抬 RPM。**

**待办（账号侧，非代码）**：找赞助方/组织者把这个 key 的账号档位升到 V1+，或拿一个高档位 key（详见与赞助方确认"赠送额度是否计入充值档位"）。

**可选代码保险（未做）**：给插话加 429 感知重试，升档前扛一扛；升档后无害。

**状态**：⏸ 搁置，等账号升档。
